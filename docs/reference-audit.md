# Portfolio Reference Audit

Research date: 2026-07-19

## Method

Each reference was reviewed in a desktop browser at 1280 x 800 and inspected for content structure, navigation, project presentation, visual hierarchy, and credibility signals. The goal is translation, not replication. No public copy, image, code, project, personal claim, or proprietary asset from these sites will be reused.

## Cross-Reference Findings

### Patterns worth using

1. Lead with a literal role and a concrete operating promise.
2. Put the most credible, runnable work before secondary portfolio pieces.
3. Pair every project with a problem, Andrej's contribution, a visual artifact, and a result or validation target.
4. Use dedicated URLs for case studies and direct links to runnable tools or source where appropriate.
5. Let the interface demonstrate judgment: clear grouping, data quality labels, restrained motion, and useful controls.
6. Keep proof near claims. Recommendations, project screenshots, and evidence should be one interaction away.

### Patterns intentionally rejected

- Percentage-based skill meters.
- Stock-photo laptop heroes.
- Portrait-first template layouts.
- Long biographies before work evidence.
- Navigation with more destinations than the portfolio needs.
- Large technology-logo walls.
- Copied dashboards, screenshots, case studies, or personal claims.
- Background animation that consumes attention or compute.
- Cookie banners or tracking scripts where no tracking is necessary.

## Public-Site Boundary

Reference URLs are research-only and must remain inside this document. They must not appear in public portfolio pages, metadata, screenshots, datasets, or project content. All redesigned code, visuals, datasets, charts, and case-study narratives will be original to Andrej's portfolio.
